title: "Accounts"

Each user can connect multiple social accounts. Accounts store encrypted OAuth tokens and are used when publishing to identify which credentials to use per platform.

---

## List connected accounts

`GET /v1/accounts`

<CodeGroup>
```bash cURL
curl "https://api.postezi.com/v1/accounts" \
  -H "Authorization: Bearer $POSTEZI_KEY"
```

```json Response
{
  "data": [
    {
      "id": "018f4a2b-...",
      "platform": "bluesky",
      "username": "you.bsky.social",
      "connected_at": "2025-06-01T12:00:00Z"
    },
    {
      "id": "018f4a2c-...",
      "platform": "linkedin",
      "username": "Your Name",
      "connected_at": "2025-06-05T09:30:00Z"
    }
  ]
}
```
</CodeGroup>

## Connect Bluesky

`POST /v1/accounts/bluesky/connect`

Bluesky uses app passwords — not your main account password. Generate one at [bsky.app/settings/app-passwords](https://bsky.app/settings/app-passwords).

<CodeGroup>
```bash cURL
curl -X POST https://api.postezi.com/v1/accounts/bluesky/connect \
  -H "Authorization: Bearer $POSTEZI_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "handle": "you.bsky.social",
    "app_password": "xxxx-xxxx-xxxx-xxxx"
  }'
```

```json Response
{
  "data": {
    "id": "018f4a2b-...",
    "platform": "bluesky",
    "username": "you.bsky.social",
    "connected_at": "2025-06-19T10:00:00Z"
  }
}
```
</CodeGroup>

Reconnecting with the same Bluesky DID updates the stored tokens instead of creating a duplicate.

## Connect LinkedIn or Threads

`POST /v1/accounts/{platform}/connect`

LinkedIn and Threads use OAuth 2.0. The API returns an authorization URL — redirect the user there to approve access.

<CodeGroup>
```bash cURL
curl -X POST https://api.postezi.com/v1/accounts/linkedin/connect \
  -H "Authorization: Bearer $POSTEZI_KEY"
```

```json Response
{
  "data": {
    "auth_url": "https://www.linkedin.com/oauth/v2/authorization?..."
  }
}
```
</CodeGroup>

After the user approves, LinkedIn redirects to the Postezi OAuth callback which stores the tokens and redirects to the dashboard.

<Note>
  Connect LinkedIn and Threads via the dashboard's **Accounts** page — the OAuth redirect flow requires a browser.
</Note>

## Connect X (Twitter)

`POST /v1/accounts/x/connect`

X uses OAuth 1.0a with your own developer credentials (BYOK). Provide all four values from your X developer app. See the [X setup guide](/dashboard/x-twitter) for how to obtain them.

Postezi verifies the credentials against the X API before storing. If verification fails, a `400` is returned.

<CodeGroup>
```bash cURL
curl -X POST https://api.postezi.com/v1/accounts/x/connect \
  -H "Authorization: Bearer $POSTEZI_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "api_key": "your_x_api_key",
    "api_secret": "your_x_api_secret",
    "access_token": "your_x_access_token",
    "access_token_secret": "your_x_access_token_secret"
  }'
```

```json Response
{
  "data": {
    "id": "018f4a2d-...",
    "platform": "x",
    "username": "yourhandle",
    "connected_at": "2025-06-25T10:00:00Z"
  }
}
```
</CodeGroup>

Reconnecting with credentials for the same X user ID updates the stored credentials rather than creating a duplicate account.

<Note>
  Your credentials are stored encrypted (AES-256-GCM) and never exposed in API responses. All four values are required — partial credentials will be rejected.
</Note>

## Disconnect an account

`DELETE /v1/accounts/{id}`

Permanently removes the account and its stored tokens. Any future publish attempts to that platform will fail until reconnected.

```bash
curl -X DELETE "https://api.postezi.com/v1/accounts/018f4a2b-..." \
  -H "Authorization: Bearer $POSTEZI_KEY"
```

Returns `204 No Content` on success.
